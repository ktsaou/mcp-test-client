# 2026-04-25 — recovering from v1.0 + adopting the maintainer framework

## 1. Working session opens

**Costa, relaying user reports — v1.0 fails the basics.** Seven points,
verbatim:

1. UX/styling primitive — no button styling, no selected-tab indicator,
   confusing.
2. "Saving forms is the worst UX ever" — `window.prompt()` for naming a
   canned request.
3. UI is tiny and hard to work with.
4. Screen real-estate badly split. Tools list takes 1/3 of screen width and
   isn't resizable.
5. Log is unmanageable: no nav between requests/responses, no copy-as-JSON
   per message, no save-to-file. JSON should respect newlines (the original
   pretty-printer did this; the port doesn't). On narrow screens timestamp +
   direction icon waste width.
6. No tooltips anywhere.
7. Tool descriptions are unreadable (truncated or hidden).

Costa's framing: "the application seems a throw away, not practical for any
reasonable use" and "you have failed".

**The expectations these violated.**

- That a "release" means the visible surface has been honestly evaluated by
  the maintainer as a user, not just `npm test && git tag`.
- That basic interaction patterns (tooltips, active states, modals over
  prompts, resizable panes, copy buttons) are table stakes, not "polish".
- That the maintainer makes decisions and judges quality — does not ship
  the first thing that compiles.

## 2. Decisions taken

- [DEC-000](../decisions/DEC-000-working-framework.md) — Adopt the
  Maintainer/Worker/Advisor three-mode framework, with live `maintainer/*`
  documentation. Codified in CLAUDE.md.
- [DEC-001](../decisions/DEC-001-mantine-v9.md) — Pick Mantine **v9** as the
  UI component library; stop hand-rolling primitives. Revised from initial
  v8 pick after the framework analyst flagged v8.3.18 as the last 8.x
  release.
- [DEC-002](../decisions/DEC-002-ux-advisor-mandatory.md) — UX-critic

  surface.
- [DEC-003](../decisions/DEC-003-json-view-newlines.md) — Reimplement the

  original `legacy/json-pretty-printer.js` behaviour properly).
- [DEC-004](../decisions/DEC-004-mantine-appshell-resizable.md) — Mantine
  `<AppShell>` for chrome, `react-resizable-panels` for inner splits;
  panels persist to localStorage.
- [DEC-005](../decisions/DEC-005-bundle-budget.md) — CI bundle-size
  tripwire at 350 KB gzipped, prevents the `@mantine/code-highlight` 300 KB

- [DEC-006](../decisions/DEC-006-json-view-ships-ahead.md) — Land the JSON
  view rewrite ahead of the Mantine migration; merged in commit
  `d7f7fb2`.
- [DEC-007](../decisions/DEC-007-non-costa-issues-into-v1-1.md) — Modal
  a11y, real tabs, light-theme contrast, empty-state copy fold into v1.1

- [DEC-008](../decisions/DEC-008-maintainer-restructure.md) — Split
  `product.md` / `feedback.md` / `decisions.md` into per-purpose
  subdirectories with a skills layer.

## 5. What I (the maintainer) learnt today

Costa's methodology feedback in this session became three guardrails:

1. **One file per working day for the log, plus a stable skills layer** —
   single growing files are forgetting at scale. Codified in
   [DEC-008](../decisions/DEC-008-maintainer-restructure.md). Guardrail
   landed in [`../log/README.md`](README.md) (this directory's contract)
   and in [`../skills/feedback-folding.md`](../skills/feedback-folding.md)
   (the routing rule for new feedback).
2. **Multiple skills with explicit guardrails** — when feedback fires,
   route it to the skill that should have prevented it; if no such skill
   exists, create one in the same edit. Codified in
   [`../skills/feedback-folding.md`](../skills/feedback-folding.md);
   today's seven-point report drove guardrail updates in
   [`../skills/release-readiness.md`](../skills/release-readiness.md),
   [`../skills/ux-review.md`](../skills/ux-review.md),
   [`../skills/advisor-spawn.md`](../skills/advisor-spawn.md), and
   [`../skills/ui-surface-change.md`](../skills/ui-surface-change.md).
3. **Delegation discipline** — Costa called this out directly: "you do
   work yourself instead of delegating." In this session I personally
   wrote the bundle-budget script, the CI tweak, and the json-view
   changes while a Worker was busy on `src/ui/`. Guardrail landed in
   [`../skills/delegation.md`](../skills/delegation.md): any file outside
   `maintainer/` and `CLAUDE.md` is now Worker territory by default;
   maintainer-mode source edits require a written justification in the
   daily log.

## 8. New user feedback — response metrics are first-class value

**Costa, relaying user reports.** Verbatim:

> the mcp server response size, duration, and estimated are important
> metrics for users.

**My reading of "estimated":** estimated token count when the response is
fed into an LLM. MCP responses exist to be consumed by LLMs; knowing how
big a `tools/call` result is in tokens is the deciding factor when a user
plans an integration (context-window budgeting). "Bytes" answers transport
cost; "duration" answers latency; "tokens" answers LLM cost. All three are
distinct and useful.

**The expectation it violated.** Browser-only MCP tools are how a developer
_evaluates_ a public server. Without these three metrics they can answer
"does it work" but not "is it usable in my system". The
[`../product/quality-bar.md`](../product/quality-bar.md) quality bar today
lists "JSON renders so a human can read it" but does not list "the user
can see how big / fast / costly the response is." Updating the bar in this
session.

**Where the fix lives.**

- [`../product/quality-bar.md`](../product/quality-bar.md) gains a new
  bullet: "Per-response metrics — bytes, duration, estimated tokens — are
  visible everywhere a response is shown."
- [DEC-009](../decisions/DEC-009-response-metrics.md) opens for the
  implementation choice (heuristic vs real tokenizer; where to surface).
  Status: decision-pending, advisor pass required before code is written.

**Release scoping.** v1.1 is gating on its UX-critic pass right now and
the migration is large. Adding scope mid-flight repeats the chaos pattern
[`../skills/release-readiness.md`](../skills/release-readiness.md) is
trying to prevent. **Decision: this ships as v1.1.1, the very next
release after v1.1 lands.** Worker brief will be drafted after v1.1's
critic clears and v1.1 is deployed.

**Falsifier.** If a user runs through the §3 flow on the v1.1.1 deploy
and cannot answer "how many tokens / how many bytes / how long" for the
last response without opening DevTools, the fix wasn't enough.

## 9. UX-critic verdict on the v1.1 candidate (dev build) — NO SHIP

Critic ran against `http://localhost:5173/` (commit `ec7f6ee`). Full report
at `/tmp/ux-review-2026-04-25-v11/` plus 35 screenshots.

**Verdict.** No-ship. Three blockers; everything else solid.

**Blockers (must fix before push):**

1. **Mobile viewport (390 × 844) is unusable.** Four panes share the
   width; REQUEST clips to "REQUES"; header subtitle overlaps title.
   `AppShell` `flex-direction` is set on `<main>` but the inner panel
   container ignores it. Needs a responsive collapse to a single-pane
   stacked layout below ~768 px (sidebar → drawer; inventory / request /
   log as tabs).
2. **Share link drops the tool call.** [`../values.md`](../values.md)
   promises "share a reproducible tool call". Today the link round-trips
   server registration only — no selected tool, no method, no args. Look
   at `src/share-url/encode.ts` (encodes them) vs
   `src/ui/share-url-loader.tsx` (only materialises the server, doesn't
   apply `tool` / `args` / `raw`). Fix is in the loader.
3. **Enter does not submit the Add Server modal.** Quality-bar item 8
   requires it. Mantine `Modal` ships focus-trap + Esc but Enter-submit
   needs a `<form onSubmit>` wrapper and a `type="submit"` primary
   button. Same pattern needed in the Save-as modal.

**Borderlines (track, do not block v1.1):**

- Empty-state copy is technically present but still one-liners — the
  [`../decisions/DEC-007-non-costa-issues-into-v1-1.md`](../decisions/DEC-007-non-costa-issues-into-v1-1.md)
  promise was "paragraph that helps".
- Light-theme log timestamps `rgb(149,149,149)` on white = ~2.85:1,
  fails WCAG AA. Borderline on quality-bar item 10.
- ARIA `valuemin (63) > aria-valuemax (35)` on the main/log separator —
  inverted, screen readers will report a broken range.
- Heap at 30 messages = 43 → 127 MB (+83 MB, ~2.7 MB/pair). At 200
  messages projects to ~270 MB. Not catastrophic; close. Already
  deferred to v1.1.1 by
  [`../decisions/DEC-007-non-costa-issues-into-v1-1.md`](../decisions/DEC-007-non-costa-issues-into-v1-1.md).

These four fold into v1.1.1 alongside the response-metrics work
([DEC-009](../decisions/DEC-009-response-metrics.md)).

**What already feels good — don't regress.** Newline-respecting JSON
view with `↵` glyphs ([DEC-003](../decisions/DEC-003-json-view-newlines.md)
falsifier passes); per-message log hover actions with `aria-label`s;
real `role="tab"` tabs with arrow-key nav; modal focus-trap; bearer
token round-trips correctly into `Authorization` header.

**Critic's prompt-improvement note.** I should have asked them to verify
**share-link state-restoration scope** explicitly: `(a)` server only,
`(b)` server + selected tool, `(c)` server + tool + method + args. The
current implementation silently shipped (a) while the values.md promise
is (c). **Guardrail:** updated [`../skills/ux-review.md`](../skills/ux-review.md)

- the UX-critic template in [`../agents.md`](../agents.md) to spell out
  share-link scope on every future pass.
